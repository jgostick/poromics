# Plan: Add RunPod Serverless Queue + Ephemeral Pod Lifecycle for Taichi Permeability

## Goal

Add a `taichi-runpod-serverless` queue backed by RunPod Serverless, and replace the existing
persistent-pod approach with a fully ephemeral pod lifecycle on `taichi-runpod-pod`. Each job
creates a fresh pod, runs computation, then immediately terminates the pod. The serverless queue
supports concurrent jobs and scales to zero; the pod queue runs jobs serially, one pod at a time.
Users pay a premium for serverless (3.0 credits/M voxels vs 2.0 for pod).

---

## Context Summary

- **Old pod flow**: Celery task → wake *persistent* pod via `RunPodOrchestrationHook` → HTTP POST
  to `taichi_server.py` → poll HTTP GET → return result. Pod stays alive between jobs.
- **New pod flow (ephemeral)**: Celery task → create *fresh* pod → wait for `RUNNING` → build
  proxy URL → HTTP POST to `taichi_server.py` → poll → return result → **terminate pod** (in
  `finally`). Pod is destroyed after every job.
- **New serverless flow**: Celery task → submit payload via RunPod Serverless REST API
  (`POST /v2/{endpoint_id}/run`) → poll (`GET /v2/{endpoint_id}/status/{job_id}`) → return result.
  No pod lifecycle managed by the task at all.
- **Endpoint routing** is already abstracted through `queue_catalog.get_queue_endpoint()` and
  `RunPodQueueMapping` DB overrides. For the ephemeral pod path the endpoint URL is constructed
  at runtime from the newly created pod ID; for the serverless path it is an endpoint_id resolved
  from settings.
- **Pricing** is already queue-driven via `queues.yaml` and `get_queue_pricing_rate()`.

---

## Decisions

| Topic | Decision |
|---|---|
| Existing queue rename | `taichi-runpod` → `taichi-runpod-pod` |
| New queue name | `taichi-runpod-serverless` |
| Pod lifecycle | Ephemeral: create fresh pod per job, terminate immediately after |
| Serverless handler location | `taichi_serverless_handler.py` at repo root |
| Pod pricing (unchanged) | 2.0 credits/M voxels |
| Serverless pricing | 3.0 credits/M voxels |

## Implementation Steps

### Step 1 — Rename `taichi-runpod` → `taichi-runpod-pod` in `config/queues.yaml`

```yaml
- name: taichi-runpod-pod          # was: taichi-runpod
  backend_key: gpu
  compute_system: taichi
  analyses: [permeability]
  endpoint_url: ""                  # dynamically constructed at runtime per job
  enabled: true
  pricing: { default_credits_per_million_voxels: "2.0000" }
```

**Side effects of rename** (all must be addressed):
- Any `RunPodQueueMapping` DB rows with `queue_name="taichi-runpod"` can be deleted — they are no
  longer used for the ephemeral pod path (no persistent pod_id to track).
- `RUNPOD_QUEUE_POD_IDS` env var entries referencing `taichi-runpod=<pod-id>` can be removed;
  the new flow uses `RUNPOD_POD_QUEUE_SPECS` instead (see Step 3).
- Any `TAICHI_QUEUE_ENDPOINTS` env overrides referencing the old name must be updated or removed.

### Step 2 — Add `taichi-runpod-serverless` queue in `config/queues.yaml`

```yaml
- name: taichi-runpod-serverless
  backend_key: serverless
  compute_system: taichi
  analyses: [permeability]
  endpoint_url: ""                  # runtime-resolved from RUNPOD_SERVERLESS_QUEUE_ENDPOINT_IDS
  enabled: true
  pricing: { default_credits_per_million_voxels: "3.0000" }
```

The `backend_key: serverless` is a new key that:
1. Populates `TAICHI_BACKEND_QUEUE_MAP["serverless"] = "taichi-runpod-serverless"`.
2. Signals to task routing that this queue uses the serverless path (no pod management).

### Step 4 — Rewrite `apps/pore_analysis/runpod_orchestration.py`

Replace the wake-up logic entirely with an ephemeral pod lifecycle.

#### Old responsibilities (remove)
- `RunPodOrchestrationHook.ensure_pod_exists` — woke a persistent pod via `resume_pod(pod_id)`
- `_queue_pod_ids()` — read `RUNPOD_QUEUE_POD_IDS` / DB mapping

#### New responsibilities
```python
def create_ephemeral_pod(queue_name: str) -> tuple[str, str]:
    """
    Create a fresh RunPod pod for the given queue.
    Resolves pod spec from settings.RUNPOD_POD_QUEUE_SPECS[queue_name].
    Polls until pod status == RUNNING (up to RUNPOD_POD_STARTUP_TIMEOUT_SECONDS).
    Returns (pod_id, endpoint_url).
    endpoint_url is constructed as:
        https://{pod_id}-{RUNPOD_POD_TAICHI_PORT}.proxy.runpod.net
    Raises RunPodTransientError on timeout or startup failure.
    """

def terminate_ephemeral_pod(pod_id: str) -> None:
    """
    Terminate a pod created by create_ephemeral_pod.
    Swallows RunPodNotFoundError (already gone). Logs other errors but does not raise,
    so callers can safely use this in finally blocks.
    """
```

Public API kept for compatibility (but semantics change):
```python
def maybe_ensure_runpod_pod(queue_name, analysis_type, endpoint_url):
    # Now a no-op stub — callers should switch to create_ephemeral_pod instead.
    # Kept temporarily to avoid hard cut-over in tasks.py.
```

### Step 5 — New `apps/utils/runpod_serverless.py`

A new module parallel to `apps/utils/runpod_pods.py`. Wraps the RunPod Serverless REST API.

```python
def submit_job(endpoint_id: str, payload: dict) -> str:
    """POST /v2/{endpoint_id}/run  →  returns RunPod job_id."""

def poll_status(endpoint_id: str, job_id: str) -> dict:
    """GET /v2/{endpoint_id}/status/{job_id}
    Returns dict with:
      status: "IN_QUEUE" | "IN_PROGRESS" | "COMPLETED" | "FAILED" | "CANCELLED"
      output: result dict (when COMPLETED)
      error:  error string (when FAILED)
    """

def cancel_job(endpoint_id: str, job_id: str) -> None:
    """POST /v2/{endpoint_id}/cancel/{job_id}"""
```

- Auth: `Authorization: Bearer {RUNPOD_API_KEY}` (reuses existing setting).
- Uses `httpx` with existing timeout/retry settings (`RUNPOD_CONNECT_TIMEOUT_SECONDS`, etc.).
- Typed exceptions reuse the `RunPodTransientError`, `RunPodAPIError`, etc. hierarchy from
  `apps/utils/runpod_pods.py`.

### Step 6 — New `apps/pore_analysis/runpod_serverless_client.py`

Task-layer adapter between `tasks.py` and `apps/utils/runpod_serverless.py`.

```python
def run_permeability_serverless(
    *,
    queue_name: str,
    image_array: np.ndarray,
    direction: str,
    max_iterations: int,
    tolerance: float,
    backend: str,
    voxel_size: float,
) -> dict:
    endpoint_id = _resolve_serverless_endpoint_id(queue_name)
    payload = {
        "image_npy_b64": _encode_array(image_array),   # same base64 format as taichi_client.py
        "direction": direction,
        "max_iterations": max_iterations,
        "tolerance": tolerance,
        "backend": backend,
        "voxel_size": voxel_size,
    }
    job_id = runpod_serverless.submit_job(endpoint_id, payload)
    return _poll_until_done(endpoint_id, job_id)        # raises on FAILED/CANCELLED
```

`_resolve_serverless_endpoint_id(queue_name)` checks:
1. DB `RunPodQueueMapping.endpoint_url` (repurposed as endpoint_id field for serverless queues).
2. `settings.RUNPOD_SERVERLESS_QUEUE_ENDPOINT_IDS[queue_name]`.
3. Raises `RunPodConfigurationError` if neither is set.

### Step 7 — New `taichi_serverless_handler.py` at repo root

RunPod Serverless handler. Uses the `runpod` Python SDK. Input/output format matches
`taichi_server.py`'s POST body / response body to maximise code sharing.

```python
import runpod
import numpy as np, base64, io

def handler(job: dict) -> dict:
    inp = job["input"]
    image_bytes = base64.b64decode(inp["image_npy_b64"])
    arr = np.load(io.BytesIO(image_bytes), allow_pickle=False)
    # run kabs (same logic as taichi_server.py _compute_permeability)
    from kabs import compute_permeability, solve_flow
    ...
    return {"permeability [lu^2]": ..., ...}

runpod.serverless.start({"handler": handler})
```

The Docker image uses the same base as the pod image but with `CMD ["python",
"taichi_serverless_handler.py"]` as the entrypoint instead of `taichi_server.py`.

### Step 8 — Update `apps/pore_analysis/tasks.py` (`run_permeability_job`)

```python
queue_name = _get_task_queue_name(self.request, default_queue="kabs-cpu")

if _is_serverless_queue(queue_name):
    # --- Serverless path: no pod management ---
    from .runpod_serverless_client import run_permeability_serverless
    with job.image.file.open("rb") as f:
        arr = np.load(f, allow_pickle=False)
    solution = run_permeability_serverless(
        queue_name=queue_name, image_array=arr, ...
    )

elif _is_ephemeral_pod_queue(queue_name):
    # --- Ephemeral pod path: create → run → terminate ---
    from .runpod_orchestration import create_ephemeral_pod, terminate_ephemeral_pod
    pod_id, endpoint_url = create_ephemeral_pod(queue_name)
    try:
        with job.image.file.open("rb") as f:
            arr = np.load(f, allow_pickle=False)
        from .analysis.permeability import run_kabs_permeability
        solution = run_kabs_permeability(..., endpoint_url=endpoint_url)
    finally:
        terminate_ephemeral_pod(pod_id)   # always runs, even on exception

else:
    # --- Local / CPU path (unchanged) ---
    with job.image.file.open("rb") as f:
        arr = np.load(f, allow_pickle=False)
    from .analysis.permeability import run_kabs_permeability
    solution = run_kabs_permeability(..., endpoint_url=None)
```

### Step 9 — Celery worker binding and concurrency

| Queue | Worker needs GPU? | Recommended concurrency |
|---|---|---|
| `taichi-runpod-pod` | No (HTTP dispatch only; GPU is on RunPod) | 1 (serial — one live pod at a time) |
| `taichi-runpod-serverless` | No | ≥4 (RunPod Serverless auto-scales) |
| `kabs-cpu` | Yes (local Taichi) | 1 |

**For `taichi-runpod-pod`**, `--concurrency=1` is important: it ensures only one pod is alive at
any time, which controls cost. Multiple concurrent pod jobs would spin up multiple pods
simultaneously — undesirable unless explicitly wanted.

**For `taichi-runpod-serverless`**, higher concurrency is safe since each task is just submitting
and polling over HTTP. RunPod Serverless handles GPU allocation.

### Step 10 — Orphaned pod mitigation

If a Celery worker is killed (SIGKILL / OOM) while a pod is running, the `finally` block in Step 8
will not execute, leaving an orphaned running pod. Two mitigations:

1. **Periodic cleanup Celery Beat task** (new, lightweight): runs every N minutes, calls
   `list_pods()`, identifies pods whose name matches the taichi ephemeral naming convention and
   have been running longer than `MAX_JOB_LIFETIME_MINUTES`, terminates them.
2. **RunPod pod TTL** (if supported): configure a maximum runtime on the pod spec so RunPod itself
   terminates the pod after a ceiling time.

Recommendation: implement option 1 as a simple scheduled task. Option 2 depends on RunPod API
support and should be investigated.

### Step 12 — `RunPodQueueMapping` model / admin UI

The existing `RunPodQueueMapping` model stored `(queue_name, pod_id, endpoint_url)` for runtime
overrides. With ephemeral pods, `pod_id` has no meaning for `taichi-runpod-pod`. Two options:

- **Option A** (minimal change): repurpose the model for serverless queues only — `pod_id` becomes
  `endpoint_id` (text field, rename display label). The admin UI remains useful for operators who
  want to update the serverless endpoint_id at runtime without a deploy.
- **Option B**: leave the model as-is but ignore it for pod queues; document that only serverless
  queues use it.

**Recommendation**: Option A. Update the admin form label from "Pod ID" to "Endpoint ID / Pod ID"
and add a note in the help text. No migration needed (field is already a char field).

### Step 13 — Remove dead health-check and pod-wake code

With ephemeral pods the endpoint is only reached after we have confirmed the pod is `RUNNING`, and
with serverless RunPod handles worker availability entirely. Pre-flight health checks against
remote endpoints are therefore redundant (for the non-fatal cases) or actively wrong (they would
hit a pod that doesn't exist yet for the ephemeral case). Remove them all — with one deliberate
exception for the Julia persistent server (see below).

## Open Questions / Risks

1. **Pod proxy URL format**: The assumed URL pattern is
   `https://{pod_id}-{port}.proxy.runpod.net`. Confirm this against an actual RunPod pod creation
   response, as the exact format may differ (some pods use connect URLs in the API response).
2. **RunPod Serverless endpoint**: Has the endpoint been provisioned yet? The endpoint_id is
   needed for `RUNPOD_SERVERLESS_QUEUE_ENDPOINT_IDS`.
3. **Handler Docker image**: Can the same base image as the pod use a different entrypoint for
   serverless, or does it need a separate Dockerfile?
4. **Polling timeout for serverless**: RunPod Serverless jobs may queue during high demand.
   Add a dedicated `RUNPOD_SERVERLESS_JOB_TIMEOUT_SECONDS` setting.
5. **Cost exposure during task failure between pod creation and computation**: The pod will
   incur billing from creation to termination even if the analysis itself fails. This is
   unavoidable with on-demand pods. The `finally` block minimises the window; document this for
   operators.
6. **`backend_key` collision**: Both the local CPU path (`kabs-cpu`, `backend_key: cpu`) and the
   ephemeral pod queue (`taichi-runpod-pod`, `backend_key: gpu`) may overlap with other queue
   keys. The `_is_ephemeral_pod_queue` helper should match on queue name or a dedicated
   `backend_key` value (e.g., `runpod-gpu`) rather than the generic `gpu` string, to avoid
   future ambiguity.
